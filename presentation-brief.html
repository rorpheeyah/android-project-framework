The Problem

Conditional Logic Sprawl

Every new market means adding branches to every existing feature. Regression risk grows as O(banks × features).

TransferViewModel.kt — the anti-pattern
if (bank == "KH") { … }         // ← scattered across every flow
else if (bank == "VN") { … }    // ← scattered across every flow
else if (bank == "PPC") { … }   // ← scattered across every flow
else { /* MY, TH, ID, … */ }

// Regression risk = O(banks × features)
Approach we rejectedCost
if (bank == "X") chainsOne PR touches every flow — merge conflicts, hidden regressions
One APK per bankCherry-picks across N branches; CI/store-listing matrix explodes
Runtime DI hot-swapCostly machinery for a flow users never actually take
The Solution

A Layered Module DAG

8 module shapes with a strict dependency DAG. Depend upward, never sideways. Forbidden imports enforced at compile time.

↑
Arrows mean "depends on"
Siblings :data, :variants-*, :features, :features-chatbot, :features-{name} never import each other. :core and :design-system are peers — neither depends on the other. All forbidden imports are enforced at compile time via Gradle module boundaries.
Four Promises

What the architecture guarantees

Not aspirational — every promise is enforced by a mechanical invariant at compile or boot time.

1
Server-side demux
One backend handles per-user routing. No KhBakongApi on the client. The Fintech*Api family is unified — the server dispatches per auth token.
2
Login-once binding
LoggedInComponent is built once at login with the active variant's policy bindings, then dropped at logout. No runtime swap, no purge sequence.
3
MG bootstrap
Only the MG URL is hardcoded. Real backend URLs, maintenance state, and version floors come from MG at cold start — no production.json baked into the binary.
4
Additive variant onboarding
New variant = new module + 2 lines. Zero changes to :features, :data, :design-system, or sibling variants. Compile-time enforced.

@HiltViewModel
class TransferInputViewModel @Inject constructor(
    private val policy: TransferAmountPolicy,  // :core
    private val repo:   TransferRepository,     // :core
) : MviViewModel<…>() {

    private fun onSubmit() = viewModelScope.launch {
        when (val v = policy.validate(state.value.amount)) {
            is Invalid -> setState { copy(error = v.reason) }
            Valid      -> repo.submit(state.value.intent)
                              .fold(…)
        }
    }
}

Design Decision

No :domain module · no use cases

:core already owns contracts and models. Variability lives in :variants-*/policy/. A use-case layer adds classes without adding flexibility.

SubmitTransferUseCase.kt
class SubmitTransferUseCase @Inject constructor(
    private val policy: TransferAmountPolicy,
    private val repo:   TransferRepository,
) {
    suspend operator fun invoke(
        intent: TransferIntent
    ): Result<TransferReceipt> =
        when (val v = policy.validate(intent.amount)) {
            is Invalid -> Result.failure(ValidationException(v))
            Valid      -> repo.submit(intent)
        }
}

Tradeoffs

What we accept — and why

📦Slightly larger APK
Variants are ~10 files each. Compiling all into one binary adds negligible size. Split-APK delivery handles store distribution if it matters.
📐Convention-based feature boundaries
:features uses package-level boundaries, not Gradle modules per screen. Lint enforces the contract. Build-performance wins justify this — see docs/14-build-performance.md.
🔄Variant change = logout → login
Runtime DI swap is costly machinery for a flow that doesn't exist in banking apps. Login-once binding is the correct model for this domain.
FAQ

Common questions

Can a user log in to different banks on the same device?▾
Yes. Logout drops the session graph (LoggedInComponent, prefs, cache) and pops navigation to root. The next login rebuilds with whatever variantId the server returns. No special code path.
What about per-bank branding (logos, colors)?▾
Future-roadmap. Branding is intentionally not in RuntimeConfig. Per-variant theming is a separate mechanism that does not affect any existing module boundaries.
What if our next project isn't multi-tenant?▾
Drop :variants-{id}. Everything else — :aos-core, :core, :design-system, :data, :features, MG bootstrap, LoggedInComponent — is reusable as a clean modular Android stack.
What if a variant grows to need its own UI, API, or screens?▾
Goes in a sibling module: :features-{feature-name} (e.g. :features-bakong-disputes), gated by a VariantCapabilities flag. The variant module stays pure — policies + DI only.
Where do use cases / interactors live?▾
There are none. ViewModel composes a :core policy + a :core repository directly — typically 5–15 lines. A *UseCase wrapper adds a class to maintain with zero new flexibility.
Why no separate :domain module?▾
:core already owns contracts + domain models. Variability lives in :variants-*/policy/. Splitting into a :domain module adds indirection without adding isolation — the forbidden import rules already enforce that :features can't reach :data or :variants-*.
