Hilt, Demystified

Components, Scoping, and the @Binds annotation — the three concepts that separate "Hilt compiles" from "Hilt actually models your app's lifetimes correctly." Grounded in the Bizplay IPPP framework's custom LoggedInComponent design.

Section 1

Hilt Components & the Hierarchy

A Hilt Component is a dependency graph with a lifetime. Hilt generates one component per Android container (Application, Activity, Fragment, ViewModel, Service) and wires them into a tree. Where a binding "lives" is decided by which component it's installed in — and that decides when it's created, when it's destroyed, and who can inject it.

The Built-In Hierarchy

Hilt creates these components automatically. Each child can see everything its parents expose; parents cannot see into children.

SingletonComponent
Application
Created in Application.onCreate(), lives until the process dies. Home of @Singleton.
▼
ActivityRetainedComponent
Survives Configuration Changes
Lives across rotation. Parent of ViewModelComponent. Scope: @ActivityRetainedScoped.
▼
ViewModelComponent
Per-ViewModel
One per @HiltViewModel instance. Scope: @ViewModelScoped.
▼
ActivityComponent
Activity Lifecycle
Dies on onDestroy(); recreated on rotation. Scope: @ActivityScoped.
▼
FragmentComponent / ViewComponent / ServiceComponent
Leaves of the tree
Bound to Fragment / View / Service lifecycles respectively.

How Dependencies Flow Down the Tree

A binding in a parent is visible to all descendants. So when you write:

NetworkModule.kt — installed in SingletonComponent
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {

    @Provides @Singleton
    fun okHttp(): OkHttpClient = OkHttpClient.Builder().build()
}

...the same OkHttpClient instance can be injected into a @HiltViewModel, an Activity, a Service, or anything else — because all of those live inside child components of SingletonComponent.

●
Rule of thumb: install bindings at the highest sensible component.Installing a binding lower in the tree restricts who can see it. Don't push everything to SingletonComponent just because it's easy — match the install scope to the actual lifetime you want.

Custom Components: When the Built-Ins Aren't Enough

The Android lifecycle doesn't always match your app's logical lifecycle. This project is a prime example — a corporate-expense app where many objects (repositories, encryption keys, the active session, variant policies) should live exactly as long as the user is logged in. There's no built-in Hilt component for "logged-in session." So this project defines a custom component:

:app/di/LoggedInComponent.kt — custom session-scoped component
// 1. Define the scope annotation
@Scope
@Retention(AnnotationRetention.RUNTIME)
annotation class LoggedInScoped

// 2. Define the component, parent = SingletonComponent
@LoggedInScoped
@DefineComponent(parent = SingletonComponent::class)
interface LoggedInComponent {
    @DefineComponent.Builder
    interface Builder {
        fun session(@BindsInstance session: Session): Builder
        fun variant(@BindsInstance variant: VariantContext): Builder
        fun tenant(@BindsInstance tenant: TenantContext): Builder
        fun build(): LoggedInComponent
    }
}

This component is built once at login (in LoggedInComponentManager) and dropped at logout — every @LoggedInScoped instance becomes GC-eligible. That's the Hilt mechanism powering the framework's "one variant + one tenant, bound at login" invariant.

Section 2

Scoping Annotations

A scope tells Hilt: "the same instance, every time, for the lifetime of this component." Without a scope, Hilt creates a new instance on every injection. Choosing the right scope is choosing the right lifetime — get it wrong and you'll either thrash GC with unnecessary allocations or leak memory by holding state too long.

Unscoped vs Scoped

UNSCOPED — new instance per inject
@Provides
fun requestIdGenerator(): RequestIdGenerator {
    return RequestIdGenerator()
}

// Every @Inject of RequestIdGenerator
// gets a fresh instance. Fine for
// stateless helpers or factories.
SCOPED — one shared instance
@Provides @Singleton
fun okHttp(): OkHttpClient {
    return OkHttpClient.Builder().build()
}

// One OkHttpClient for the whole
// process. Connection pool, cache,
// and threadpool are shared.

When to Scope — And When Not To

SCOPE IT
Expensive to construct
Retrofit, OkHttp, Room — anything wrapping a thread pool, native handle, or large allocation.
SCOPE IT
Holds shared state
In-memory caches, session state, observable flows that consumers must agree on.
SCOPE IT
Identity matters
A repository whose internal StateFlow must be the same instance for every collector.
DON'T SCOPE
Stateless & cheap
Mappers, formatters, request builders. Constructing them is essentially free.
DON'T SCOPE
Short-lived helpers
If lifetime doesn't matter, don't bind one — let Hilt make a new one and let it die.
DON'T SCOPE
Just to "be safe"
Scope is a memory contract, not a perf optimization. Wrong scope is worse than no scope.

Memory Leaks: The Cost of Over-Scoping

The classic Hilt mistake: injecting a short-lived dependency into a long-lived scope. The long-lived scope holds the reference; the short-lived object can never be collected.

⚠
Anti-pattern: leaking an Activity Context into a Singleton Asking Hilt to provide @ActivityContext Context into a @Singleton binding will fail to compile — Hilt catches it. But the same mistake is possible at the architecture level: scoping a class that holds a Context-bearing field as @Singleton will silently leak that Activity for the whole process. Audit the transitive fields of every @Singleton.
⚠
Anti-pattern: scoping per-session state to @Singleton In this project, scoping the Session or AuthToken as @Singleton would mean the credentials survive logout — a security regression. That's why the framework defines @LoggedInScoped: a scope tied to the login lifetime, not the process lifetime. Drop the component → all sensitive references become GC-eligible immediately.
✓
Right-sized scoping checklist
  1. What's the longest this instance should live?
  2. Pick the smallest component whose lifetime matches.
  3. Audit transitive state — does it hold anything bigger than itself?
  4. If unsure, leave it unscoped. You can always scope it later.
Section 3

@Binds vs @Provides

Both annotations tell Hilt how to satisfy a dependency request. The difference is mechanical at the source level but significant at compile time — @Binds generates strictly less code and runs strictly faster.

The Mental Model

  • @Provides — "Here's a function. Call it to build the instance." Used when you need to construct something — call a builder, configure it, return it.
  • @Binds — "When someone asks for this interface, give them this implementation." Used when you have a class with an @Inject constructor() and just need to tell Hilt which interface it satisfies.

Side-by-Side: The Exact Syntax Difference

Same goal — bind AuthRepository (interface) to IpppAuthRepo (implementation). Two ways to write it:

@Provides — works, but more code
@Module
@InstallIn(SingletonComponent::class)
object AuthModule {

    @Provides
    @Singleton
    fun authRepository(
        api: IpppAuthApi,
        session: SessionStore,
    ): AuthRepository {
        return IpppAuthRepo(api, session)
    }
}

// Module is `object` (or has companion).
// Function manually calls the constructor.
// You must list every dependency by hand.
@Binds — preferred for interfaces
@Module
@InstallIn(SingletonComponent::class)
abstract class AuthModule {

    @Binds
    @Singleton
    abstract fun authRepository(
        impl: IpppAuthRepo,
    ): AuthRepository
}

// Module is `abstract class`.
// Function is `abstract` — no body.
// IpppAuthRepo must have @Inject constructor.
// Hilt wires the constructor params for you.

When to Use Which

Use @Binds when…Use @Provides when…
You're binding an interface to an implementation You're providing a type you don't own (Retrofit, OkHttpClient, Room)
The implementation has an @Inject constructor() Construction requires builder configuration or factory calls
You want minimal boilerplate You need conditional logic or computed defaults
You're using multibindings (@IntoMap, @IntoSet) with interface impls You're providing primitives or values from a config

Why Prefer @Binds?

  1. Less generated code. @Provides generates a factory class with a method that calls your function. @Binds generates a typecast — that's it. Smaller APK, faster build.
  2. Less to maintain. If IpppAuthRepo's constructor signature changes (a new parameter is added), the @Binds version does not need to be touched. With @Provides, you'd update the function signature too.
  3. Catches dead code. If the implementation doesn't have an @Inject constructor, @Binds fails to compile — forcing you to be explicit about how the impl is constructable.

The Three Rules of @Binds

●
  1. The module must be an abstract class (not object).
  2. The function must be abstract and take exactly one parameter — the implementation type.
  3. The return type must be the interface (or supertype) you're binding to.

You can mix @Binds and @Provides in the same module: declare the module as abstract class, put @Binds functions on it directly, and put @Provides functions in a companion object. This project uses exactly that pattern in DataModule — see the real-world example below.

Section 4

Real-World Code Example

A complete, production-shaped example: an Activity injects a logger interface, whose implementation lives in a separate file and is wired in via @Binds. This is the canonical Hilt + interfaces + Activity pattern.

Step 1 — The interface and implementation

analytics/AnalyticsTracker.kt
interface AnalyticsTracker {
    fun track(event: String, params: Map<String, Any> = emptyMap())
    fun screen(name: String)
}
analytics/FirebaseAnalyticsTracker.kt
import com.google.firebase.analytics.FirebaseAnalytics
import javax.inject.Inject
import javax.inject.Singleton

@Singleton
class FirebaseAnalyticsTracker @Inject constructor(
    private val firebase: FirebaseAnalytics,
) : AnalyticsTracker {

    override fun track(event: String, params: Map<String, Any>) {
        firebase.logEvent(event, params.toBundle())
    }

    override fun screen(name: String) {
        firebase.logEvent(FirebaseAnalytics.Event.SCREEN_VIEW, bundleOf(
            FirebaseAnalytics.Param.SCREEN_NAME to name,
        ))
    }
}

Step 2 — The Hilt module with @Binds

analytics/AnalyticsModule.kt
import com.google.firebase.Firebase
import com.google.firebase.analytics.FirebaseAnalytics
import com.google.firebase.analytics.analytics
import dagger.Binds
import dagger.Module
import dagger.Provides
import dagger.hilt.InstallIn
import dagger.hilt.components.SingletonComponent
import javax.inject.Singleton

@Module
@InstallIn(SingletonComponent::class)
abstract class AnalyticsModule {

    // @Binds for the interface → impl edge.
    // The impl's constructor is wired automatically.
    @Binds
    @Singleton
    abstract fun bindAnalyticsTracker(
        impl: FirebaseAnalyticsTracker,
    ): AnalyticsTracker

    companion object {
        // @Provides for the SDK type we don't own.
        @Provides
        @Singleton
        fun firebaseAnalytics(): FirebaseAnalytics = Firebase.analytics
    }
}

Both patterns coexist in one module: @Binds for the interface binding, and a companion object with @Provides for the SDK-owned type (FirebaseAnalytics) we can't annotate with @Inject.

Step 3 — Inject into an Activity

ui/SettingsActivity.kt
import android.os.Bundle
import androidx.activity.ComponentActivity
import dagger.hilt.android.AndroidEntryPoint
import javax.inject.Inject

@AndroidEntryPoint
class SettingsActivity : ComponentActivity() {

    @Inject lateinit var analytics: AnalyticsTracker

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        analytics.screen("settings")
        // …rest of onCreate
    }
}

The Activity depends on the interface, not the implementation. @AndroidEntryPoint tells Hilt to generate the field-injection plumbing. The implementation is swappable in tests: replace the @Binds in a test module with a fake tracker, and SettingsActivity never knows the difference.

✓
Why this is the canonical shape The Activity sees one type (the interface). The wiring lives in one module. Construction is implicit (via @Inject constructor). Adding a parameter to FirebaseAnalyticsTracker requires zero changes to AnalyticsModule or SettingsActivity. That's Hilt earning its keep.
Section 5

How This Project Uses Hilt

The Bizplay IPPP framework leans heavily on Hilt — not just for plumbing, but as the mechanism that enforces the architecture. Three patterns are worth highlighting.

1. Custom LoggedInComponent for Session-Scoped State

The framework's load-bearing invariant: the variant and tenant bind once at login and stay. Hilt's built-in components don't model "logged-in session," so the project defines a custom one. Every per-session binding (repositories, the active VariantContext, the TenantContext, encryption keys) is installed in LoggedInComponent with @LoggedInScoped. On logout, LoggedInComponentManager.drop() releases the component — every scoped instance becomes GC-eligible at once.

:data/di/DataModule.kt — repository bindings (@Binds + @Provides combined)
@Module
@InstallIn(LoggedInComponent::class)
abstract class DataModule {

    @Binds @LoggedInScoped
    abstract fun authRepo(impl: IpppAuthRepo): AuthRepository

    @Binds @LoggedInScoped
    abstract fun receiptRepo(impl: IpppReceiptRepo): ReceiptRepository

    @Binds @LoggedInScoped
    abstract fun approvalRepo(impl: IpppApprovalRepo): ApprovalRepository

    // …and four more — one @Binds per :core repository interface

    companion object {
        @Provides @LoggedInScoped
        fun ipppRetrofit(
            factory: RetrofitFactory,
            config: RuntimeConfig,
        ): Retrofit = factory.builder(config.urls.main).build()

        @Provides @LoggedInScoped
        fun authApi(retrofit: Retrofit): IpppAuthApi =
            retrofit.create(IpppAuthApi::class.java)

        // …and six more *Api creates
    }
}

The interface-to-impl wiring uses @Binds (the repos have @Inject constructors). The Retrofit instances use @Provides because we need to call retrofit.create() ourselves.

2. Multibindings as the Single Point of Variant Dispatch

The hardest architectural rule in this project: no code branches on variantId or tenantId for dispatch. Every variant binds the same set of :core policy interfaces. Plain @Binds would error on duplicate bindings, so the framework uses Dagger multibindings keyed on variant ID:

:variants-kr/di/KrVariantModule.kt
@Module
@InstallIn(LoggedInComponent::class)
abstract class KrVariantModule {

    @Binds @IntoMap @VariantKey("kr") @LoggedInScoped
    abstract fun amountPolicy(impl: KrExpenseAmountPolicy): ExpenseAmountPolicy

    @Binds @IntoMap @VariantKey("kr") @LoggedInScoped
    abstract fun receiptRenderer(impl: KrReceiptRenderer): ReceiptRenderer

    // …one @IntoMap @VariantKey("kr") binding per :core policy interface
}

Each variant's module binds its own implementations into the same maps. A small VariantResolverModule in :app does one lookup — map[session.variant.id] — and exposes the result as the live binding. Adding a new variant requires zero changes anywhere else. Hilt does the dispatch via type, not if/when.

3. @InstallIn Enforces the Module Topology

The architecture forbids :features from depending on :data or :variants-* — the Logic-Blind rule. Hilt makes this both visible and enforceable: every binding declares @InstallIn(LoggedInComponent::class), so features inject :core interfaces and Hilt resolves them at compile time from whichever module provided the impl. The feature module never imports the concrete class — it doesn't even know it exists.

●
The big idea Hilt isn't just plumbing in this project — it's the architectural firewall. Custom components model the right lifetimes (LoggedInComponent = login session). Multibindings replace branching on identifiers. @Binds keeps the boilerplate-to-signal ratio low across dozens of repository interfaces. Get these three right and the rest of the codebase follows.
