SECTION 1

Hilt Components & Hierarchy

A Hilt Component is a generated container that knows how to construct and serve a fixed set of bindings. Each component is anchored to a real Android lifecycle — it is created when that lifecycle begins, and destroyed when the lifecycle ends. Bindings flow down the tree (children see ancestors), never up.

SingletonComponent            // created in Application.onCreate()
│                              // destroyed when the process dies
│                              scope: @Singleton
│
├── ActivityRetainedComponent  // survives configuration changes
│   │                          scope: @ActivityRetainedScoped
│   │
│   ├── ViewModelComponent    // one per ViewModel instance
│   │                          scope: @ViewModelScoped
│   │
│   └── ActivityComponent     // rebuilt on rotation
│       │                      scope: @ActivityScoped
│       │
│       ├── FragmentComponent // lives with the Fragment
│       │                      scope: @FragmentScoped
│       │
│       └── ViewComponent     // custom Views with @WithFragmentBindings
│                              scope: @ViewScoped
│
└── ServiceComponent           // bound to Service lifecycle
                               scope: @ServiceScoped
↓
Reading the arrows
ActivityComponent can inject anything bound in SingletonComponent and ActivityRetainedComponent — but the reverse is not true. Children can see ancestors; ancestors cannot reach into children. This is why a Singleton may never depend on an Activity-scoped binding.
●SingletonComponent@Singleton
Process-lifetime container. Anchor for OkHttp, Retrofit, Room, DataStore, app-wide logging, and any other infrastructure that should outlive every screen.
●ActivityRetainedComponent@ActivityRetainedScoped
Survives configuration changes (rotation, dark-mode switch). The parent of ViewModelComponent. Useful for state that must outlive a single Activity instance but not the process.
●ViewModelComponent@ViewModelScoped
One per @HiltViewModel. Lives as long as the ViewModel — ideal for per-screen caches or use-case helpers that should not be shared across screens.
●ActivityComponent@ActivityScoped
Rebuilt on every configuration change. Holds Activity-aware collaborators: navigators that need an @ActivityContext, snackbar hosts, permission helpers.
●FragmentComponent@FragmentScoped
Tied to the Fragment lifecycle. Rarely the right choice in Compose-first projects — most Compass code skips Fragments entirely.
●ServiceComponent@ServiceScoped
Bound to a Service. Use for foreground-service collaborators (e.g. background sync workers that aren't WorkManager).
★
Custom components are allowed
Hilt's built-in tree covers ~95% of apps. When a feature has a lifecycle Hilt does not model — for Compass, that is the logged-in session — you define a custom component with @DefineComponent. The Compass LoggedInComponent is a child of SingletonComponent that is built at login and dropped at logout, with the custom scope @LoggedInScoped.
SECTION 2

Scoping Annotations

A scope annotation tells Hilt: "cache this instance for the lifetime of the matching component". Without a scope, every injection point gets a new instance — which is sometimes exactly what you want, and sometimes a silent bug.

ScopeComponentBehaviorUse for
(none)—New instance per injection point. Stateless by definition.Mappers, formatters, validators, pure functions wrapped in a class.
@SingletonSingletonComponentOne instance for the entire process.OkHttpClient, Retrofit, Room, DataStore, Json.
@ActivityRetainedScopedActivityRetainedComponentOne per Activity, survives rotation.State holders that outlive a single Activity instance.
@ViewModelScopedViewModelComponentOne per @HiltViewModel.Per-screen caches, use-case helpers used only by one ViewModel.
@ActivityScopedActivityComponentOne per Activity, rebuilt on rotation.Activity-context-bound helpers (snackbars, navigators).
@FragmentScopedFragmentComponentOne per Fragment instance.Rare in Compose-first stacks.
@LoggedInScoped
(custom — Compass)		LoggedInComponentOne per login session. Dropped on logout.Session-bound repos, variant policies, Session.
⚠
The memory-leak trap
A binding scoped to a long-lived component must never hold a reference to a short-lived one. The classic foot-gun: marking a class @Singleton while it captures an Activity in its constructor. The Activity will be retained for the entire process — that is a leak. Rule of thumb: if a class holds anything Context-flavoured, prefer the narrowest scope that satisfies the use case. When unsure, leave it unscoped — a new instance per call is cheap, leaks are not.
i
Scope is not a performance optimization
A common anti-pattern is sprinkling @Singleton "to avoid creating objects". Don't. Scope is a correctness contract about identity: "this instance must be the same across these injection sites". If the answer is "doesn't matter", leave it unscoped.
SECTION 3

@Provides vs @Binds

Both annotations teach Hilt how to satisfy a type. The choice between them is mechanical once you know the rule: use @Binds when an interface implementation is constructor-injectable; use @Provides when you have to build the instance yourself.

▸@Providesalways-works
A regular function inside an object module. You write the body; Hilt calls it whenever the return type is needed. Required for: third-party types, builder-pattern construction, Retrofit.create() calls, conditional logic, anything Hilt cannot constructor-inject.
▸@Bindsinterface-mapping
An abstract function inside an abstract class module that maps an interface to a constructor-injectable implementation. Hilt generates the body at compile time — there is no runtime cost, and the bytecode is smaller than the equivalent @Provides.

Side-by-side: the exact same wiring, two ways

@Provides — explicit, more code
AnalyticsModule.kt
@Module
@InstallIn(SingletonComponent::class)
object AnalyticsModule {

    @Provides
    @Singleton
    fun provideAnalytics(
        impl: FirebaseAnalyticsTracker
    ): AnalyticsTracker = impl
}
Works, but noisy: Hilt could have figured this out itself. The function body is just a passthrough.
@Binds — idiomatic, generated
AnalyticsModule.kt
@Module
@InstallIn(SingletonComponent::class)
abstract class AnalyticsModule {

    @Binds
    @Singleton
    abstract fun bindAnalytics(
        impl: FirebaseAnalyticsTracker
    ): AnalyticsTracker
}
Hilt sees: "FirebaseAnalyticsTracker is the impl for AnalyticsTracker." That's the whole contract.

The implementation that makes @Binds possible

FirebaseAnalyticsTracker.kt
// :data/analytics/FirebaseAnalyticsTracker.kt
internal class FirebaseAnalyticsTracker @Inject constructor(
    private val firebase: FirebaseAnalytics,
    private val session: Session,
) : AnalyticsTracker {

    override fun track(event: AnalyticsEvent) {
        firebase.logEvent(event.name, event.params.toBundle())
    }
}
i
The deciding question
"Can the implementation be constructor-injected?" If yes, use @Binds — it is more concise, generates less code, and clearly states "this interface maps to that class". If no (third-party class, complex builder, conditional logic), use @Provides.
Aspect@Provides@Binds
Module typeobject (or class)abstract class (or interface)
Function typeRegular, with a bodyabstract, no body
Generated codeCalls your function at runtimeHilt generates the binding directly
Works for third-party typesYesNo (impl must be @Inject-constructable)
Works for interfacesYes (but verbose)Yes (preferred)
Builder/factory constructionYesNo
Bytecode sizeLargerSmaller
Mixable in the same moduleYes — but only if the module is abstract and the @Provides functions are companion object members.
SECTION 4

Real-World Example

A production-shape vertical slice: an Activity injects an interface, an @Binds module maps it to the impl, and the impl is constructor-injected with its own dependencies. This is the pattern you write 90% of the time.

1 · The contract — an interface

:core/analytics/AnalyticsTracker.kt
package com.compass.core.analytics

interface AnalyticsTracker {
    fun track(event: AnalyticsEvent)
    fun identify(userId: String)
}

data class AnalyticsEvent(
    val name: String,
    val params: Map<String, Any> = emptyMap(),
)

2 · The implementation — constructor-injected

:data/analytics/FirebaseAnalyticsTracker.kt
package com.compass.data.analytics

internal class FirebaseAnalyticsTracker @Inject constructor(
    private val firebase: FirebaseAnalytics,
    private val clock: Clock,
) : AnalyticsTracker {

    override fun track(event: AnalyticsEvent) {
        val params = Bundle().apply {
            event.params.forEach { (k, v) -> putString(k, v.toString()) }
            putLong("ts", clock.now().toEpochMilli())
        }
        firebase.logEvent(event.name, params)
    }

    override fun identify(userId: String) {
        firebase.setUserId(userId)
    }
}
i
Note the internal visibility
The impl is internal — the rest of the codebase only sees the AnalyticsTracker interface. This is the Compass convention: in :data and :variants-*, only the Hilt module is intentionally public.

3 · The Hilt module — @Binds + a @Provides for the third-party dep

:data/di/AnalyticsModule.kt
package com.compass.data.di

@Module
@InstallIn(SingletonComponent::class)
abstract class AnalyticsModule {

    // Interface → impl mapping. Pure compile-time wiring.
    @Binds
    @Singleton
    abstract fun bindAnalytics(
        impl: FirebaseAnalyticsTracker
    ): AnalyticsTracker

    companion object {

        // Third-party type — must use @Provides because Firebase
        // is a Google SDK class we cannot annotate with @Inject.
        @Provides
        @Singleton
        fun provideFirebase(
            @ApplicationContext ctx: Context
        ): FirebaseAnalytics = FirebaseAnalytics.getInstance(ctx)
    }
}

4 · The Activity consumes the interface

:app/MainActivity.kt
package com.compass.app

@AndroidEntryPoint
class MainActivity : ComponentActivity() {

    @Inject lateinit var analytics: AnalyticsTracker   // the interface, not the impl

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        analytics.track(AnalyticsEvent(name = "app_open"))

        setContent { CompassApp() }
    }
}
★
Why this pattern earns its keep
The Activity depends on an interface in :core, not on Firebase. To swap Firebase for Mixpanel, write a MixpanelAnalyticsTracker, change one line in the module, and ship. To test the Activity, supply a fake AnalyticsTracker with @TestInstallIn. Neither change touches the call sites — that is the entire point of DI, and @Binds is the cheapest way to express it.
SECTION 5

Hilt in the Compass Framework

This repository is a documentation specification of the Compass Framework — a multi-region Android banking architecture. Hilt is the load-bearing wiring mechanism. Below is a tour of how the patterns from sections 1–4 actually compose in the framework.

@LoggedInScopedA custom session-scoped component
Compass defines LoggedInComponent as a child of SingletonComponent with the custom scope @LoggedInScoped. It is built once by BootCoordinator after a successful login, holds Session + VariantContext + TenantContext, and is dropped by LogoutHandler at logout. Everything @LoggedInScoped becomes GC-eligible the moment the component is dropped — there is no purge sequence.
@InstallInWhere each module installs
:aos-core infrastructure (OkHttp, Retrofit factory, security primitives) → SingletonComponent. The :data repository implementations and every :variants-* policy module → LoggedInComponent. Hilt auto-discovers them at compile time — there is no central registry of modules.
@Binds + @IntoMapMultibindings as the single point of dispatch
Every :variants-{id} binds the same set of :core policy interfaces. Plain @Binds would fail with a duplicate-binding error. The framework's solution: @Binds @IntoMap @VariantKey("kh") @LoggedInScoped. A single resolver in :app/di/VariantResolverModule.kt picks the active variant's entry — no when (variantId) branches anywhere else.
KSPAnnotation processing
Compass uses KSP (Kotlin Symbol Processing) for Hilt, not kapt. KSP is roughly 2× faster on cold builds and significantly faster on incremental builds — see docs/14-build-performance.md.

The variant binding pattern, verbatim from docs/07-variants.md

:variants-kh/di/KhVariantModule.kt
@Module
@InstallIn(LoggedInComponent::class)
abstract class KhVariantModule {

    @Binds @IntoMap @VariantKey("kh") @LoggedInScoped
    abstract fun amountPolicy(impl: KhTransferAmountPolicy): TransferAmountPolicy

    @Binds @IntoMap @VariantKey("kh") @LoggedInScoped
    abstract fun feeCalc(impl: KhFeeCalculator): FeeCalculator

    @Binds @IntoMap @VariantKey("kh") @LoggedInScoped
    abstract fun capabilities(impl: KhCapabilities): VariantCapabilities

    // … one @Binds per :core policy interface
}
:app/di/VariantResolverModule.kt
@Module
@InstallIn(LoggedInComponent::class)
object VariantResolverModule {

    @Provides @LoggedInScoped
    fun transferAmountPolicy(
        variant: VariantContext,
        all: Map<String, @JvmSuppressWildcards TransferAmountPolicy>,
    ): TransferAmountPolicy = checkNotNull(all[variant.id.value]) {
        "No TransferAmountPolicy registered for variant ${variant.id}"
    }

    // … one provider per :core policy interface
}
i
Notice the @Binds / @Provides division
The variant module uses @Binds because each impl is constructor-injectable. The resolver in :app uses @Provides because the body is non-trivial — it picks an entry from a map. Same module file pattern, two annotations, each used where it fits.

The orchestrator (:app)

:app/CompassApplication.kt
@HiltAndroidApp
class CompassApplication : Application() {
    // Hilt's annotation processor wires every @Module @InstallIn(*)
    // at compile time. No central registry needed.
}
▸Onboarding a new variant
Drop in a variants-{id}/ module with @Binds @IntoMap @VariantKey("{id}") @LoggedInScoped for each :core policy interface. Add one line to settings.gradle and one entry to VariantCatalogue. Zero edits to :features, :data, :design-system, or sibling variants. Hilt's compile-time aggregation does the rest.
▸Tenants share the pattern
Structural TenantPolicy impls bind with @Binds @IntoMap @TenantKey("{id}") @LoggedInScoped — the same multibinding shape, one level deeper than variants. Non-structural (flag/param) tenant differences need no Hilt entry at all; ViewModels just read tenant.flags.*.
SUMMARY

Key Takeaways

1Components are lifecycles
Pick the component that matches how long the binding should live. Custom components (like Compass's LoggedInComponent) exist for lifecycles the standard tree does not model.
2Scopes are correctness, not perf
Scope expresses "same instance everywhere within this lifecycle". When in doubt, leave it unscoped — the cost is trivial, leaks are not.
3Default to @Binds
If the impl is constructor-injectable, use @Binds. Reach for @Provides only when you must — third-party types, builders, conditional construction.
4Depend on interfaces
Inject the contract, never the concrete class. The Compass framework's "Logic-Blind" :features module is the same principle applied at architectural scale.
