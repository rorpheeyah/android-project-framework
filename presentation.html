The Challenge

Conditional Logic
Sprawl

The existing Bizplay4.0_IPPP codebase has roughly 124 call sites of DetailConfig.isXxx() scattered across Activities, Fragments, ViewModels, and adapters. Every new corporate customer (POSCO, Lotte, NIA, Shinsegae, ITCen, WIPS, HANA, IBS, SPC, …) multiplies the test surface of every existing flow.

💀
O(tenants × features) regression risk
Every new corporate customer multiplies the testing surface of every existing screen, exponentially.
🕸️
Every PR touches every flow
Scattered DetailConfig.isNIA() chains mean a NIA-only field-visibility change must be reviewed across every screen that branches.
🔒
New tenant = APK release
Today's hardcoded compile-time tenant predicates require shipping a new binary to onboard a customer.
✅
Framework: polymorphism does the dispatch
Zero if(tenant.id==…) anywhere. Server-sourced TenantFlags. New tenant = three files inside its variant.

Industry Parallels

Why This Works
in Production

The Bizplay framework borrows from battle-tested patterns at scale. These analogs prove the model: large multi-tenant and multi-jurisdiction apps that serve dozens of customers / regions from a single binary use the same polymorphic dispatch + module-per-customer-axis pattern.

▲
Wise (TransferWise)
50+ countries · one app
Wise's mobile app serves remittance corridors with wildly different KYC, AML, and limit regimes — a single binary chooses a "corridor policy" by user residency at sign-in, never branching in UI.
Bizplay uses VariantContext + @VariantKey for the same login-time selection.
U
Uber RIBs
city-specific business rules
Uber's RIBs architecture isolates each business capability behind interactors that swap implementations per city/country (surge pricing, regulator caps, payment rails) — no concrete city ID ever leaks into UI logic.
Bizplay :variants-{id} mirrors RIBs' per-region interactor swap.
♪
Spotify Backstage
plugin-as-polymorphism
Backstage treats every integration as a plugin behind a typed contract — the host never branches on which plugin is active. Adding a new vendor is strictly additive, exactly like adding a Bizplay variant or tenant.
Bizplay :features ↔ :variants-* = host ↔ plugin.
◆
Clean Architecture
dependency-rule enforcement
Robert Martin's dependency rule: source-code dependencies point only inward toward higher-level policy. Bizplay enforces it at compile time via Gradle: :features cannot import :variants-* or :data.
Compile errors > code review for invariants.
A
Google's Now in Android
official modularization sample
Google's reference modular Android app splits :core, :data, :feature, :design-system with the same dependency directions Bizplay uses — but does not address multi-tenant variance.
Bizplay extends Now-in-Android with variant + tenant axes.
$
Salesforce / SAP B2B SaaS
multi-tenant enterprise apps
Enterprise SaaS platforms serve thousands of corporate customers from one codebase — using typed per-tenant feature flags and policy injection rather than per-customer compile-time branches. The pattern scales to hundreds of tenants.
Identical shape: typed TenantFlags, server-sourced values, no hardcoded predicates.

Lifecycle

Boot Phases

Variant + tenant selected once at login — no in-session swap, no purge sequence. Only the MgGate URL is hardcoded in the binary.

🔑 Only one URL is hardcoded: the MgGate URL per build type (today's Conf.SITE_MG_URL + "/MgGate"). Every other network endpoint, maintenance state, version floor, and approval-form URL comes from MgGate's RuntimeConfig at cold start.
UI Architecture

MVI Pattern

Strict unidirectional data flow. Every screen in :features follows this contract — no exceptions, no short-cuts. Replaces today's MVVM-with-OnComTranListener callbacks.

UiState
ViewModel → View
StateFlow<S>
  • Immutable snapshot of what's on screen
  • Form values · loading / error indicators
  • Capability-derived booleans (showOcrButton)
  • Tenant-flag-derived booleans (showEmployeeId)
  • Never holds one-shot nav commands
UiEvent
View → ViewModel
onEvent(E)
  • Each user interaction — tap, text change
  • Exhaustive when(event) in ViewModel
  • Each event maps to exactly one handler
  • No ambient side effects from View side
UiEffect
ViewModel → View
SharedFlow<F>
  • One-shot — navigation, toasts, snackbars
  • replay=0 → no replay on screen rotation
  • extraBufferCapacity=16 → no drop
  • Same effect twice → two deliveries (≠ state)
ViewComposable
──[UiEvent]──▶
ViewModelMviViewModel<S,E,F>
──[UiState]──▶
Viewre-renders
View also collects UiEffect (SharedFlow) for one-shot side effects — navigation never lives in UiState
Multi-Axis Design
