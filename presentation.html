The Challenge

Conditional Logic
Sprawl

A single Android binary must serve multiple companies, regions, and customer organizations — chosen at login, not install. Without architecture, the codebase collapses under variant-specific branching.

💀
O(variants × features) regression risk
Every new region multiplies the testing surface of every existing feature, exponentially.
🕸️
Every PR touches every flow
Scattered if-chains mean a KH fee-rule change must be reviewed across all regions and tenants.
🔒
New variant = global code review
No isolation means onboarding VN affects KH's tested code paths — a compliance nightmare.
✅
Compass: polymorphism does the dispatch
Zero if(variantId==…) anywhere. New variant = new module, zero :features edits.
❌ Anti-pattern — TransferViewModel.kt
// Scattered across every ViewModel, Repo, Validator
fun validateAmount(amount: Money) {
  if (variantId == "kh") {
    limit = KHR_4_000_000
    fee = khFeeCalc.calculate(amount)
  } else if (variantId == "vn") {
    limit = VND_500_000_000
    fee = vnFeeCalc.calculate(amount)
  } else if (variantId == "kr") {
    // 11 tenants × many checks …
    if (tenantId == "nia") hideEmployeeId()
    if (tenantId == "wips") clearIdOnApproval()
    if (tenantId == "shinsegae") showCustomApproval()
  }
}
✅ Compass — TransferInputViewModel.kt
// ViewModel only sees :core interfaces. Zero variant knowledge.
class TransferInputViewModel @Inject constructor(
  private val repo:   TransferRepository,    // :core
  private val policy: TransferAmountPolicy, // :core
  capabilities:       VariantCapabilities      // :core
) : MviViewModel<...>() {

  fun validateAmount(amount: Money) {
    // Polymorphism routes to active variant's impl
    val result = policy.validate(amount) // ✓ no if-chains
    setState { copy(validation = result) }
  }
}
Industry Parallels

Why This Works
in Production

Compass borrows from battle-tested patterns at scale. These analogs prove the model: large fintech and consumer apps that serve dozens of jurisdictions from a single binary use the same polymorphic dispatch + module-per-region pattern.

▲
Wise (TransferWise)
50+ countries · one app
Wise's mobile app serves remittance corridors with wildly different KYC, AML, and limit regimes — a single binary chooses a "corridor policy" by user residency at sign-in, never branching in UI.
Compass uses VariantContext + @VariantKey for the same login-time selection.
U
Uber RIBs
city-specific business rules
Uber's RIBs architecture isolates each business capability behind interactors that swap implementations per city/country (surge pricing, regulator caps, payment rails) — no concrete city ID ever leaks into UI logic.
Compass :variants-{id} mirrors RIBs' per-region interactor swap.
♪
Spotify Backstage
plugin-as-polymorphism
Backstage treats every integration as a plugin behind a typed contract — the host never branches on which plugin is active. Adding a new vendor is strictly additive, exactly like adding a Compass variant.
Compass :features ↔ :variants-* = host ↔ plugin.
◆
Clean Architecture
dependency-rule enforcement
Robert Martin's dependency rule: source-code dependencies point only inward toward higher-level policy. Compass enforces it at compile time via Gradle: :features cannot import :variants-* or :data.
Compile errors > code review for invariants.
A
Google's Now in Android
official modularization sample
Google's reference modular Android app splits :core, :data, :feature, :design-system with the same dependency directions Compass uses — but does not address multi-region variance.
Compass extends Now-in-Android with variant + tenant axes.
$
Revolut · Monzo
single-app multi-jurisdiction
European neobanks ship one app across the EU, UK, US, and APAC — each with distinct compliance regimes. Public engineering posts confirm a strategy-pattern + per-region module approach to keep the UI surface stable.
Identical shape: pure-policy variant modules, shared UI engine.

UI Architecture

MVI Pattern

Strict unidirectional data flow. Every screen in :features follows this contract — no exceptions, no short-cuts.

UiState
ViewModel → View
StateFlow<S>
  • Immutable snapshot of what's on screen
  • Form values · loading / error indicators
  • Capability-derived booleans (showQrScanner)
  • Never holds one-shot nav commands
  • Never holds raw Throwable instances
UiEvent
View → ViewModel
onEvent(E)
  • Each user interaction — tap, text change
  • Exhaustive when(event) in ViewModel
  • Each event maps to exactly one handler
  • No ambient side effects from View side
UiEffect
ViewModel → View
SharedFlow<F>
  • One-shot — navigation, toasts, snackbars
  • replay=0 → no replay on screen rotation
  • extraBufferCapacity=16 → no drop
  • Same effect twice → two deliveries (≠ state)
ViewComposable
──[UiEvent]──▶
ViewModelMviViewModel<S,E,F>
──[UiState]──▶
Viewre-renders
View also collects UiEffect (SharedFlow) for one-shot side effects — navigation never lives in UiState
