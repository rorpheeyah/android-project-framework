The Problem

Conditional Logic Sprawl

The existing Bizplay4.0_IPPP codebase has ~124 call sites that branch on corporate-customer identity. Every new tenant means touching every flow. Regression risk scales as O(tenants × features).

Scattered across every flow in Bizplay4.0_IPPP
if      (DetailConfig.isNIA())           { … }  ← scattered across every flow
else if (DetailConfig.isPOSCO_ICT())   { … }  ← scattered across every flow
else if (DetailConfig.isLotte())        { … }  ← scattered across every flow
else if (DetailConfig.isWIPS())         { … }  ← scattered across every flow
else if (DetailConfig.isShinsegae())    { … }
else if (DetailConfig.isITCen())        { … }
else if (DetailConfig.isHANA())         { … }
else    { /* IBS, SPC, Chilsung, … */ }

Four Promises

What the Framework Guarantees

Every architectural decision is in service of one of these four invariants. Any PR that contradicts them is a regression.

🌐
Server-Side Demux
One IPPP backend handles per-user, per-USE_INTT_ID, per-COMPANY_CD routing. No KrKakaoPayApi on the client. The Ippp*Api family is split by feature area, not by variant.
🔑
Login-Once Binding
LoggedInComponent built at login with variant + tenant context, dropped at logout. No runtime swap. Variant or tenant change = logout → login.
🚀
MgGate Bootstrap
Only the MgGate URL is hardcoded (Conf.SITE_MG_URL + "/MgGate"). All backend URLs, maintenance state, and version floors come from MgGate at cold start via RuntimeConfig.
➕
Additive Onboarding
New variant = new module + 2 lines. New tenant = 1–3 files inside its variant. Zero change to :features or :data.
Two Axes, Three Layers

Variant ⊃ Tenant ⊃ Account

Three orthogonal bindings. Variant and tenant bind at login and never swap at runtime. Account is a session-level header flip — cheap, instantaneous.

Variant
Region / regulator boundary
:variants-kr  /  :variants-kh  /  :variants-vn — drives: amount limits, fee calculators, formatters, capability flags, support contacts, calendars, receipt rendering.
↓ login-time bind, no runtime swap
Tenant
Corporate customer boundary
POSCO ICT · Lotte · NIA · Shinsegae · ITCen · WIPS · HANA · IBS · SPC · default — drives: TenantFlags (named booleans) and TenantParams (typed fields). Structural differences use a :core/policy/ interface per-tenant impl.
↓ login-time bind, no runtime swap
Account
Institution membership — the real runtime axis
USE_INTT_ID / COMPANY_CD — today's SelectUserInttIdActivity flow. Stamped on every request by AccountIdInterceptor from Session.activeAccountId: StateFlow<AccountId>. Just a header swap — no DI rebuild, no logout required.
ℹ
Key distinction — Switching corporate accounts (the daily picker) is an account flip: cheap, same LoggedInComponent. Switching variant or tenant requires logout → login to rebuild the DI graph.

Design Decision

No :domain Module · No Use Cases

:core already plays the domain-contracts role. Variability lives in :variants-*/policy/ and TenantFlags. The orchestration that use cases would perform happens in the ViewModel — typically 5–15 lines.

Side-by-Side: Same Flow, Two Designs

❌ With a use case layer
class SubmitReceiptUseCase @Inject constructor(
    private val policy: ExpenseAmountPolicy,
    private val repo:   ReceiptRepository,
) {
    suspend operator fun invoke(
        draft: ReceiptDraft
    ): Result<Receipt> =
        when (val v = policy.validate(draft.amount)) {
            is ValidationResult.Invalid ->
                Result.failure(ValidationException(v.reason))
            ValidationResult.Valid ->
                repo.create(draft)
        }
}

class ReceiptDetailViewModel @Inject constructor(
    private val submit: SubmitReceiptUseCase,
    // hides policy + repo behind a wrapper
) : MviViewModel<…>() { … }
✅ Bizplay framework — VM composes directly
class ReceiptDetailViewModel @Inject constructor(
    private val policy: ExpenseAmountPolicy,
    private val repo:   ReceiptRepository,
) : MviViewModel<…>() {

    private fun onSubmit() = viewModelScope.launch {
        when (val v = policy.validate(state.value.amount)) {
            is ValidationResult.Invalid ->
                setState { copy(error = v.reason) }
            ValidationResult.Valid ->
                repo.create(state.value.draft).fold(…)
        }
    }
}

Tradeoffs

Costs We Accept

Every architectural choice has costs. These are the ones this framework consciously accepts in exchange for the invariants it provides.

Slightly larger APK
Every variant's Hilt module compiles into the binary. Only the entries matching the logged-in user's variantId are exercised; the others sit inert in the maps.

Why OK: Variants are ~10 files each, tenants ~3. The compile-in cost is negligible.
Convention-based feature boundaries
Feature boundaries in :features are packages, not Gradle modules.

Why OK: Lint enforces it; build-perf wins justify the trade. Heavy-SDK and variant-locked features still get their own sibling modules.
Variant or tenant change = logout-login
Changing variant/tenant mid-session requires logout → re-login.

Why OK: Runtime swap is costly machinery for a non-real flow. The institution switch (the real daily flow) is a cheap header swap.
Java → Kotlin migration over time
The existing Bizplay Java codebase stays compilable; new code is Kotlin + Compose + Hilt + MVI.

Why OK: Incremental migration is lower risk than a big-bang rewrite.
FAQ

Frequently Asked Questions

Common questions from engineering leads reviewing the architecture.

Can a user log in to different corporate accounts on the same device?
+
Yes. Logout drops the session graph (component, prefs, SQLCipher session scope, cache) and pops navigation to root. The next login rebuilds with whatever variantId + tenantId the server returns. No special code path.
What about per-tenant branding (POSCO red vs Lotte navy vs NIA orange)?
+
Future-roadmap. Branding is intentionally not in RuntimeConfig; per-tenant theming would be a separate TenantBrandPolicy mechanism that :app weaves into BizTheme composition locals.
What if a customer wants their own backend?
+
Doesn't happen — IPPP is one backend with server-side demux per USE_INTT_ID + COMPANY_CD. If a customer genuinely needs a different protocol, they're a new variant, not a new tenant.
What if a variant grows to need its own UI / API / screens?
+
Goes in a sibling module: :features-{feature-name} (e.g. :features-hipass for Korea-only highway tolls), gated by a VariantCapabilities flag. The variant module stays pure policies + DI.
Where do use cases / interactors live?
+
There are none. ViewModel composes a :core policy + a :core repository directly. The variability use cases would carry lives in :variants-*/policy/. The orchestration is 5–15 lines in the ViewModel.
Why no separate :domain module?
+
:core already owns contracts + models. Variability lives in :variants-*/policy/ and TenantFlags. Splitting across two modules buys no new isolation — it just adds another recompile trigger.
What happens to the existing DetailConfig.isXxx() chain?
+
Deleted. Each predicate maps to a named TenantFlags field (e.g. isNIA() → tenant.flags.hidesEmployeeId + requiresIdNumberCapture), or — for structural differences — a per-tenant impl of a :core/policy/ interface.
What happens to BizWebview and BrowserBridge?
+
The Java BizWebview becomes the Compose BizWebView in :aos-core/webview/. The BrowserBridge family collapses to a single-method WebActionBridge with a versioned payload. URLs come from RuntimeConfig.webRoutes, not hardcoded loadUrl strings.
